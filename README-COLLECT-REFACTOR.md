# 采集链路重构说明

## 重构立场

本次重构严格遵守以下原则：

1. 不做向后兼容
2. 允许直接调整表结构与字段语义
3. 不保留旧链路作为兜底
4. 不做过渡态双写
5. 不做历史数据平滑迁移

这不是一次“局部优化”，而是对现有采集、分类、展示链路的直接替换。

旧设计只要与新目标冲突，就直接删除、改写或废弃，不保留兼容入口。

## 问题定义

当前系统的核心问题不是实现细节，而是模型本身错误。

现有链路把以下几类数据混在一起处理：

- 采集事实数据
- 分类规则数据
- 分类投影数据
- 搜索展示数据
- 缓存与标签数据

直接后果如下：

1. 修改分类规则会回写历史展示数据
2. 主站采集结果依赖后置重建才能稳定展示
3. 分类重建会串联搜索标签重建
4. 系统长期依赖“事后修正历史数据”维持一致性
5. 数据量一大，规则修改就会触发高成本全量重算

这套设计必须整体废弃，不能继续在上面做修补。

## 重构目标

重构后的系统必须满足以下硬约束：

1. 主站采集时直接按当时的分类与规则快照生成最终展示数据
2. 历史数据一旦入库，后续修改分类或规则不得再回写历史数据
3. 新分类、新规则只对下一轮主站全量采集生效
4. 附属站只负责播放源补充，不参与分类与展示结果计算
5. 切换主站时，继续清空旧主站数据并重新同步新主站分类

一句话定义新模型：

**规则只影响未来采集，不修正历史结果。主站直接产出最终展示数据，附属站只补播放源。**

## 新模型

### 主站

主站是唯一的数据框架来源，负责：

1. 写入影片详情事实
2. 写入来源分类事实
3. 在采集当下完成分类规则匹配
4. 直接生成最终 `search_info`

主站采集完成后，数据就是最终展示结果，不再依赖后置回写修正。

### 附属站

附属站只负责：

1. 写入播放列表
2. 补充来源映射
3. 刷新播放源相关摘要

附属站禁止负责：

1. 分类匹配
2. 分类树更新
3. 分类映射重建
4. `search_info` 分类字段修正
5. 标签重建

### 规则生效机制

分类规则和分类树只在主站采集时生效。

行为约束必须固定为：

1. 采集时读取当前规则快照
2. 采集时直接计算最终分类归属
3. 采集时直接写入最终展示结果
4. 后续规则修改不影响历史数据
5. 需要新规则生效时，必须重新执行主站全量采集

这里不允许保留“规则修改后自动修正历史数据”的旧逻辑。

## 数据职责重定义

### 事实层

只保存采集事实，不承担回写修正责任。

包括：

- `movie_detail_info`
- `source_categories`

### 规则层

只保存当前规则，不负责修正历史投影。

包括：

- `mapping_rules`

### 展示层

直接保存采集当次算出的最终展示结果。

包括：

- `search_info`

`search_info` 的新语义必须明确为：

**最终展示结果表，不再是允许被后台规则反复回写的中间表。**

### 分类框架层

只表达当前主站下的分类框架，不承担历史展示数据修正责任。

包括：

- `categories`
- `category_mappings`

### 缓存层

只做缓存或附加投影，不反向驱动主链路。

包括：

- `search_tag_items`

## 强制删除的旧行为

以下行为与新模型冲突，必须直接删除，不保留兼容分支：

1. 规则增删改后自动触发 `CategoryRebuild`
2. 分类重建后批量回写历史 `search_info`
3. 为修正规则结果而全量重建搜索标签
4. 规则重载时顺带刷新分类缓存的混杂行为
5. 将 `search_info` 作为可被反复修正的中间表使用
6. 将主站展示结果依赖到事后异步回写任务

这些逻辑不是“可优化项”，而是必须移除的错误设计。

## 数据结构调整

本次重构允许直接改表，不考虑旧数据兼容。

### 必改项

`search_info` 增加：

- `category_version`
- `rule_version`

用途：

1. 标记该条数据生成时使用的分类版本
2. 标记该条数据生成时使用的规则版本
3. 作为排障和追踪依据

### 可直接同步改造项

`movie_detail_info` 可同步增加：

- `category_version`
- `rule_version`

如果主站事实与展示结果都要保留同一轮采集快照标记，就直接一起加，不做犹豫。

如果后续发现 `categories`、`category_mappings`、`search_info` 的字段定义不再适合新模型，也可以直接改字段名、删字段或重建表结构，不保留旧语义。

## 新写入流程

### 主站采集写入流程

主站采集流程应收敛为：

1. 写入 `movie_detail_info`
2. 解析来源分类
3. 按当前规则计算根分类与子分类
4. 生成最终展示字段
5. 直接写入 `search_info`
6. 刷新该影片自己的标签或相关最小影响面数据

这里不允许再存在：

1. 先写半成品，再等待异步重建修正
2. 采集完成后再跑全量分类修复
3. 通过规则修改去回补历史展示结果

### 附属站采集写入流程

附属站采集流程应收敛为：

1. 写入 `movie_playlist`
2. 维护来源映射
3. 刷新对应影片的播放摘要

附属站不得写入或触发：

- 分类树
- 分类映射
- `search_info` 分类结果
- 历史分类修正
- 全量标签重建

## 主站切换行为

主站切换逻辑保留，但语义更明确：

1. 主站提升、降级、URI 变化时，直接清空旧主站全量数据
2. 清空后直接同步新主站分类树
3. 新主站重新全量采集，生成新的事实数据与展示结果

这里不做任何兼容迁移，也不保留旧主站数据的过渡逻辑。

## 实施策略

### 第一阶段：切断旧回写链路

直接删除以下触发关系：

1. 规则变更 -> 分类全量重建
2. 分类重建 -> 历史 `search_info` 回写
3. 规则变更 -> 全量搜索标签重建

目标：先让规则修改退化为单纯配置更新。

### 第二阶段：主站采集直接固化展示结果

直接把分类匹配逻辑并入主站采集写路径。

目标：主站采集结束时，`search_info` 已经是最终结果。

### 第三阶段：收紧附属站职责

删除附属站对分类链路的一切影响。

目标：附属站只保留播放源补充职责。

### 第四阶段：清理旧模型残留

删除所有仍依赖“事后修正历史数据”的旧函数、旧任务、旧字段语义。

目标：代码里只保留新模型，不保留双轨并存。

## 验收标准

重构完成后，必须满足以下结果：

1. 主站采集完成后，展示数据可直接使用
2. 修改分类规则后，历史数据展示完全不变
3. 新规则必须通过下一轮主站全量采集才能生效
4. 附属站采集不会触发分类重算
5. 附属站采集不会改写主站展示分类结果
6. 切换主站时，旧主站数据仍会被直接清空
7. 系统不再依赖任何“规则修改后修正历史数据”的后台任务

## 最终结论

本次重构不是为旧链路补洞，而是直接换模型。

新的系统约束只有四条：

1. 主站决定框架
2. 主站采集时直接产出最终展示结果
3. 附属站只补播放源
4. 规则只影响未来采集，不影响历史结果

凡是不符合这四条的旧代码、旧表义、旧任务，都应该直接删掉或重写，不保留兼容层。
