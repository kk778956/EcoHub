# FAQ 与排障

这份文档只覆盖当前仓库的真实实现。

## 采集与数据

### 为什么必须配置主站

主站负责影片主数据骨架，包括检索索引、详情信息、分类归属和前台列表快照。没有主站时，附属站播放列表无法稳定挂载到完整影片上，前台搜索和详情也缺少基础数据。

### 主站和附属站怎么分工

- 主站写入影片主数据。
- 附属站写入播放列表。
- 详情页和播放页会把附属站播放源聚合到主站影片下。
- 后台支持对同一影片执行“更新全部站点”。

### 为什么切换主站后要重建数据

主站决定影片主数据、分类和搜索入口。切换主站后，旧主站的数据骨架不再可信，需要由新主站重新建立，避免分类、详情和列表混用旧语义。

### 为什么会做内容去重

系统会优先使用豆瓣 ID 归并同一部影片；缺失豆瓣 ID 时，再使用标题等信息生成内容指纹。这样可以减少不同采集源带来的重复详情和重复搜索结果。

### 手动停止采集会发生什么

停止会阻止继续拉取新分页，并取消仍在运行的任务。已经成功写入队列的数据会继续完成收尾发布，保证已入库内容可以被前台读到。

如果停止前已经有部分页面写入成功，前台看到的可能是这次停止前已成功入库的部分增量。

### 为什么搭建完成看不到影片

EcoHub 不内置影视数据。首次搭建完成后，需要先在后台配置采集源，并执行采集任务。

采集任务完成后，系统会把已入库影片发布到列表快照和读模型中；发布完成后，前台首页、分类、搜索、详情页、后台影片列表以及 TVBox / MacCMS 接口才能读取到这些数据。

## 快照、筛选与缓存

### 为什么采集完成后还要发布快照

前台列表、筛选和 TVBox 接口读取的是发布后的快照和读模型，而不是直接扫描采集过程中的临时变化。这样能让前台查询语义更稳定，也能减少请求时的实时聚合成本。

### 增量发布为什么仍然可能耗时

增量发布只处理 affected mids，但仍需要完成这些工作：

- 查询 `film_index` 和详情有效性。
- 删除并写入当前 active version 的列表快照。
- 删除并重建这些影片对应的筛选倒排索引。
- 事务成功后刷新内存读模型和投影索引。

大量 `mid` 一次发布时，数据库写入和读模型重建仍会占用时间。

### 为什么修改配置后前台没有立刻变化

常见原因：

- 浏览器缓存。
- CDN 或反向代理缓存。
- 当前运行实例不是最新代码。
- 修改的是未公开给前台使用的字段。
- 快照或筛选发布还没有完成。

后端会在主站切换、分类重建、快照发布后主动刷新项目自身缓存，但无法控制浏览器、CDN 或外部代理缓存。

### 为什么公共分类页和 TVBox 结果应该一致

两者共用同一套后端查询语义：

- 相同分类条件。
- 相同剧情、地区、语言、年份筛选解释。
- 相同“其他”处理逻辑。
- 相同排序语义。

如果仍不一致，优先检查缓存、运行实例和请求参数。

### “最近更新”到底看什么

最近更新只看主站资源更新时间。

这意味着：

- 主站数据变化会影响最近更新。
- 附属播放源同步不会改变最近更新排序。
- 前台分类页和 TVBox 列表排序语义一致。

## 登录与权限

### 为什么后台页面能打开，但接口提示未登录

`/manage` 页面只由前端做 cookie 存在性预检查。真正的 JWT 校验、Redis token 校验和权限判断都在后端 `/api/manage/*`。

页面能进入壳子不代表接口一定有效，最终以后端接口返回为准。

### 为什么使用 cookie 而不是 localStorage

后台登录态使用 `HttpOnly` cookie：

- 前端不需要自己保存 token。
- 后端统一控制登录态和自动续期。
- 减少 token 被前端脚本读取的风险。

### 访客账号能做什么

访客账号可以查看后台数据，但所有写操作会被后端 `WriteAccess` 中间件拒绝。

演示访客账号：`guest / guest`。

### 默认账号是否可以直接用于生产

不建议。默认账号只适合初始化和演示。正式部署后应立即修改密码，或替换为自己的账号体系。

## Docker 与部署

### Docker 里的 `API_URL` 需要手动配置吗

不需要手动配置。Compose 会按 `SERVER_PORT` 自动生成：

```env
API_URL=http://server:${SERVER_PORT:-8080}
```

`API_URL` 同时用于 Next 服务端取数和 `/api/*` rewrite。Docker 容器中的 `127.0.0.1` 指向当前容器自身，不是后端容器，所以不要在根目录 `.env` 中配置它。

### Docker 环境连不上宿主机数据库怎么办

优先检查：

- `MYSQL_HOST` / `REDIS_HOST` 是否是容器可访问地址。
- Linux 环境下 `host.docker.internal` 是否可用。
- 数据库用户是否允许容器网络来源连接。
- 宿主机防火墙是否放通端口。
- MySQL / Redis 是否只监听了 `127.0.0.1`。

### 使用内置 MySQL / Redis 启动失败怎么办

优先检查：

- 宿主机 `3306` 或 `6379` 是否已被占用。
- `docker compose logs -f mysql`。
- `docker compose logs -f redis`。
- `server.environment` 中的数据库、Redis 密码是否和服务配置一致。
- 内置 MySQL / Redis 默认不对宿主机暴露端口，不会占用宿主机 `3306` / `6379`。

### 前端打开后接口全部失败怎么办

优先检查：

- `server` 容器是否健康。
- `web` 容器中的 `API_URL` 是否与 `SERVER_PORT` 一致。
- 反向代理是否正确转发 `/api/*`。
- 浏览器请求是否被 HTTPS、CORS 或 cookie 策略拦截。

### `web` 和 `server` 为什么会端口冲突

本地开发约定：

- `server` 默认监听 `8080`。
- `web` 默认监听 `3000`。

如果手动把 Next 改到 `8080`，就会和后端冲突。修改 `web/.env.local` 中的 `PORT` 后需要重启前端服务。

## 已知注意项

- `/api/config/basic` 当前仍是公开接口。
- 前端 lint 可能仍有图片优化相关 warning。
- Compose 是否启动内置 MySQL / Redis 取决于启动命令。
- 大批量快照发布仍会聚合 affected snapshots 后统一刷新读模型，极大数据量下会有内存峰值。
